# Project Guide: Next.js + FastAPI + Supabase

This file is the source of truth. The same rules are reflected in `.cursor/rules/` for Cursor.

## Overview

Full-stack app: Next.js (frontend), FastAPI (backend), Supabase (database/auth). Orchestration: Makefile for local dev, Docker for containerization.

## Tech Stack

- **Frontend:** Next.js 15+ (App Router), Tailwind CSS, TypeScript. Path alias: `@/` → `frontend/src/`.
- **Backend:** FastAPI (Python 3.12+), Pydantic v2, Supabase-py. Entrypoint: `app.main:app`.
- **Database/Auth:** Supabase (PostgreSQL).

## Architecture & Conventions

- **Type safety:** Frontend TypeScript types in `frontend/src/types/` must mirror Backend Pydantic models in `backend/app/models/`. No manual guessing of types.
- **Naming:** CamelCase for React components; snake_case for Python functions and variables.
- **Guardrails:** Do not rely on or modify `backend/.venv/`, `node_modules/`, or `frontend/.next/` unless the task explicitly requires it.
- When researching libraries or fixing bugs, use the **context7** MCP server to verify current documentation and breaking changes.

## Development Commands

- Install all: `make install`
- Full stack: `make dev`
- Frontend only: `make frontend`
- Backend only: `make backend`
- Docker: `docker compose up --build` / `docker compose down`

## Frontend (Next.js App Router)

- **Server-first:** Use Server Components by default. Add `"use client"` only where needed for interactivity. Do not put `"use client"` on top-level `page.tsx` or `layout.tsx`.
- **Next.js 15:** `params`, `searchParams`, and `cookies` are Promises — always `await` them before use.
- **Data fetching:** Use `fetch()`. Do not use axios. Use Next.js caching (e.g. `next: { revalidate: ... }`).
- **Robustness:** Each main route segment should have `loading.tsx` and `error.tsx`.
- **Styling:** Tailwind only. Reuse variables from `frontend/src/app/globals.css` for radii and colors.
- **Imports:** Use the `@/` alias; avoid deep relative paths (e.g. `../../`).
- **Components:** PascalCase. Reusable UI in `frontend/src/components/`. Do not add Framer Motion or other animation libs unless they are in project dependencies.
- **API routes** (`frontend/src/app/api/`): When adding validation, use **zod** for schemas; define and export types with `z.infer<typeof schema>`.
- **API communication:** Frontend calls backend via `/api/v1` (proxied or absolute URL). Provide TypeScript interfaces for API responses.

---

## Backend (FastAPI)

- **Entrypoint:** `backend/app/main.py` — run with `uvicorn app.main:app --reload --port 8000`.
- **Structure:** Routers in `backend/app/routers/` (mount with prefix `/api/v1`). Models in `backend/app/models/`.
- **Strict typing:** Do not return raw `dict`, `Any`, or untyped JSON. Every endpoint must declare a Pydantic `response_model`.
- **Async:** Use `async def` for routes and DB work; avoid blocking I/O.
- **Dependencies:** Use `Annotated` for DI, e.g. `db: Annotated[Client, Depends(get_supabase_client)]`. Use `get_supabase_client()` from `app.db`; do not instantiate the Supabase client inside route handlers.
- **CORS:** Configured for `http://localhost:3000`; extend `allow_origins` only when adding new frontend origins.

## Supabase

- **Frontend (public):** `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` — client-side only, for RLS-restricted access.
- **Backend:** `SUPABASE_URL`, `SUPABASE_ANON_KEY` via `get_supabase_client()` in `app.db`. `SUPABASE_SERVICE_ROLE_KEY` is backend-only; never expose it or prefix it with `NEXT_PUBLIC_`.
- **Auth:** Use Supabase Auth Helpers/SSR in Next.js. Protect routes via Next.js middleware/SSR, not by calling `supabase.auth.getSession()` in every `page.tsx`.
- **RLS:** Enable RLS on every table. If queries return empty unexpectedly, check SQL policies first.
- **RBAC:** Use JWT claims or a `profiles` table in RLS policies; do not use different API keys for user tiers.
- **Backend ops:** Use the centralized `app.db` client. When acting on behalf of a user, prefer the user’s JWT so RLS applies; avoid defaulting to the service role.
- **Schema changes:** When proposing new tables or schema changes, provide the corresponding `CREATE POLICY` SQL for `anon`, `authenticated`, and `admin` as relevant.

## Git & PR Workflow

- **Pre-commit:** Before committing, verify the build (e.g. `npm run build` in frontend, `pytest` in backend if applicable). Do not commit broken code.
- **Commits:** One logical change per commit. Use conventional commits: `feat:`, `fix:`, `docs:`, `refactor:`.
- **After a successful task and verification:** Generate a conventional commit message and commit; then create a PR with a descriptive title and body. Do not wait for explicit permission for small, non-breaking changes.
- **PR body:** Include **Summary** (what and why) and **Test plan** (concrete steps to verify, e.g. “Run `make dev`, log in, check `/dashboard`”).
- **Fallback:** If the repo is not a git repo or `gh` is not available, output the suggested commit message and PR title/description for the user to run manually.
